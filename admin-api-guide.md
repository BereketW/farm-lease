# FarmLease Admin API Guide

This guide details the core REST APIs available to System Administrators. As an Admin, you have global oversight over all Clusters, Proposals, Agreements, and Payments. You possess exclusive rights to verify clusters and can step in to resolve or verify payment receipts and agreement terms.

## ­ƒöÉ Authentication

All API requests must include a valid session cookie generated by Better Auth.
- Endpoint: `http://localhost:3000/api/*`
- Ensure you log in using an account with the `ADMIN` role.

## ­ƒÅÿ´©Å Clusters API

Admins are responsible for reviewing pending cluster registrations and either verifying them (making them public for investors) or rejecting them.

### 1. List All Clusters
Retrieve a list of all clusters across the system, regardless of their status.

- **URL:** `GET /api/clusters`
- **Query Parameters:**
  - `region` (string): Filter by region.
  - `search` (string): Search by name, description, or location.
  - `cropType` (string): Filter by a specific crop.
  - `minSize` / `maxSize` (number): Filter by total area in hectares.
- **Admin Visibility:** Admins see all `PENDING`, `VERIFIED`, and `REJECTED` clusters.
- **Response:**
  ```json
  {
    "clusters": [
      {
        "id": "uuid",
        "name": "Hawassa Farmers Union",
        "status": "PENDING",
        "totalArea": "150.5",
        "region": "Sidama",
        // ...
      }
    ]
  }
  ```

### 2. View Cluster Details
View the full details of a cluster, including its geospatial boundaries, supporting documents, and farmer roster.

- **URL:** `GET /api/clusters/:id`
- **Response:** Returns the full `cluster` object including nested `representatives` and `farmers`.

### 3. Verify or Reject a Cluster (Admin Only)
Once a Representative registers a cluster, it sits in `PENDING` status. An Admin must review the provided `documents` and `geodata` and make a decision.

- **URL:** `POST /api/clusters/:id/decision`
- **Body:**
  ```json
  {
    "decision": "VERIFY", // or "REJECT"
    "reason": "Documents verified against national registry." // Optional, required context if rejecting
  }
  ```
- **Action:** 
  - If `VERIFY`, status becomes `VERIFIED` and the cluster becomes visible to Investors.
  - If `REJECT`, status becomes `REJECTED` and the Representative must start a new registration.
- **Response:** Returns the updated `cluster` object.

## ­ƒôä Proposals API

Admins have read-only oversight over the negotiation pipeline. They do not draft or accept proposals, but they can audit the entire conversation.

### 1. List All Proposals
- **URL:** `GET /api/proposals`
- **Query Parameters:**
  - `status` (string): e.g., `DRAFT`, `SUBMITTED`, `UNDER_NEGOTIATION`, `ACCEPTED`.
  - `clusterId` (string): Filter proposals by a specific cluster.
- **Admin Visibility:** By passing no filters, an Admin retrieves the entire negotiation history across the platform.

### 2. View Proposal & Negotiation Chat
- **URL:** `GET /api/proposals/:id`
- **Response:** Returns the proposal details, terms, and the array of `messages` (the negotiation chat log).

## ­ƒñØ Agreements & Payments API

Once a proposal is accepted, an Agreement is automatically generated. Admins have powerful override capabilities here.

### 1. Edit Agreement Terms (Admin Override)
If a dispute arises, an Admin can force an update to the terms or clauses of an agreement.
*Note: Editing terms automatically wipes all existing signatures and resets the agreement to `DRAFT` status.*

- **URL:** `PATCH /api/agreements/:id/terms`
- **Body:**
  ```json
  {
    "terms": { /* updated terms JSON */ },
    "clauses": [
      { "title": "Clause 1", "body": "Updated clause text..." }
    ],
    "note": "Admin intervention: adjusted budget per offline mediation."
  }
  ```

### 2. Cancel an Agreement (Admin Override)
Admins can forcefully terminate an active or drafting agreement.

- **URL:** `POST /api/agreements/:id/cancel`
- **Body:**
  ```json
  {
    "reason": "Fraudulent activity detected."
  }
  ```

### 3. Verify Payment Receipts (Admin or Rep)
Investors upload offline payment receipts (bank slips). A Representative usually verifies these, but an Admin can step in to approve or reject them.

- **URL:** `POST /api/receipts/:id/decision`
- **Body:**
  ```json
  {
    "decision": "VERIFY", // or "REJECT"
    "reason": "Payment confirmed in corporate bank statement."
  }
  ```
- **Action:** If `VERIFY` is sent, the associated Agreement automatically moves from `PENDING_SIGNATURES` to `ACTIVE`.

## ­ƒøí´©Å Audit Logs (Internal)

Because Admins have destructive and verification capabilities, **every Admin action is strictly logged in the database**.

When an Admin verifies a cluster, edits an agreement, or rejects a receipt, the `logAudit` function writes an immutable record to the `audit_logs` table containing:
- `actorId` (The Admin's ID)
- `action` (e.g., `"VERIFY"`, `"TERMS_EDITED"`)
- `targetType` (e.g., `"Cluster"`)
- `targetId`
- `details` (Includes the reason provided during the action)
