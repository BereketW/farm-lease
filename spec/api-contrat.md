# FarmLease API Contract

Last updated: 2026-05-02

Recent changes (2026-05-01/02): Added an internal `EmailLog` model and mailer persistence. The DB migration `packages/db/prisma/migrations/20260501101355_add_email_logs/` is present and the mailer writes `EmailLog` rows; there is no public `/api/email-logs` endpoint.

This document specifies the HTTP contract currently exposed by the codebase. It is derived from the live Express routes in `apps/server` and the Next.js auth proxy in `apps/web`.

## 1) API surface overview

### Base URLs

- Backend API: `http://localhost:3000`
- Frontend app: `http://localhost:3001`
- Auth proxy on the web app: `http://localhost:3001/api/auth/*`

### Mounted route groups

- `GET /health`
- `/api/clusters`
- `/api/users`
- `/api/proposals`
- `/api/agreements`
- `/api/receipts`
- `/api/notifications`
- `/api/uploads` (imported in server bootstrap; currently not present as a module in the tree)
- `/api/auth/*` (Better Auth handler mounted by the Next.js app)

## 2) Shared contract rules

### 2.1 Authentication

- Most routes require a valid session.
- Express routes use Better Auth session cookies via `getRequestSession(req)` or `requireSession`.
- In development, `x-dev-user-id` can impersonate a user and overrides cookie identity in supported paths.
- Routes protected by `requireRole(...)` also enforce the corresponding Prisma role.

### 2.2 Common headers

- `Content-Type: application/json` for JSON requests.
- `x-dev-user-id` is accepted in development for impersonation.
- Multipart upload routes use `multipart/form-data`.

### 2.3 Response shape conventions

- List endpoints return an object wrapper, not raw arrays.
- Detail endpoints generally return an object keyed by the resource name.
- Mutations usually return the mutated row in an object wrapper such as `{ proposal }` or `{ agreement }`.
- `PATCH /api/notifications/read-all` returns `204 No Content`.

### 2.4 Error format

The code currently emits simple JSON error objects:

```json
{ "error": "ERROR_CODE" }
```

Some validation errors add `issues`, and some state errors add contextual fields such as `status`, `required`, or `missing`.

Common error codes observed in the codebase:

- `UNAUTHENTICATED`
- `FORBIDDEN`
- `INVALID_BODY`
- `INVALID_QUERY`
- `NOT_FOUND`
- `CLUSTER_NOT_FOUND`
- `AGREEMENT_NOT_FOUND`
- `INVALID_STATE`
- `ONLY_INVESTOR`
- `ONLY_REPRESENTATIVE`
- `ONLY_REP_OR_ADMIN`
- `ONLY_INVESTOR_CAN_UPLOAD`
- `RECEIPT_FILE_REQUIRED`
- `INCOMPLETE_DRAFT`
- `ALREADY_SIGNED`

### 2.5 Pagination and limits

- `/api/clusters` returns up to 100 rows.
- `/api/proposals` returns up to 100 rows.
- `/api/agreements` returns up to 100 rows.
- `/api/notifications` returns up to 50 rows.
- `/api/proposals/:id/messages` supports cursor pagination with `before` and `limit`.
- Message `limit` defaults to 50 and is capped at 100.

### 2.6 IDs and timestamps

- IDs are UUID strings generated by Prisma.
- Timestamps are ISO 8601 strings in JSON responses.
- Decimal fields are serialized as strings in API responses.

## 3) Health check

### `GET /health`

Purpose: liveness probe for server uptime.

Auth: none.

Response:

```json
{ "status": "ok" }
```

## 4) Auth contract

### `GET /api/auth/*` and `POST /api/auth/*`

These routes are not hand-authored in `apps/server`; they are mounted by `apps/web/app/api/auth/[...all]/route.ts` and re-export the Better Auth handler from `packages/auth`.

Contract notes:

- The auth API handles Better Auth flows such as sign-in, sign-up, session fetch, and sign-out.
- The exact subroutes and payloads are owned by Better Auth, not by custom repo code.
- The client wrapper in `packages/auth/src/client.ts` points to this mounted route tree.

## 6) Users API

File owner: `apps/server/src/modules/users/routes.ts`

### 6.1 `POST /api/users/setup-profile`

Auth: required.

Purpose: set the active role after registration and create the matching role profile record.

Body:

```json
{ "role": "INVESTOR" }
```

Body schema:

- `role` must be one of `INVESTOR | FARMER | REPRESENTATIVE | ADMIN`

Response:

```json
{ "ok": true, "role": "INVESTOR" }
```

Side effects:

- Updates `user.role`.
- Marks `user.status` as `ACTIVE`.
- Upserts the matching role profile:
    - `InvestorProfile`
    - `FarmerProfile`
    - `RepresentativeProfile`

Errors:

- `401 UNAUTHENTICATED`
- `400 INVALID_BODY`

- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`

### 7.5 `POST /api/proposals`

Auth: required.

Role: `INVESTOR` only.

Purpose: create and immediately submit a proposal.

Body:

```json
{
    "clusterId": "uuid",
    "budget": 50000,
    "durationMonths": 12,
    "startDate": "2026-05-01T00:00:00.000Z",
    "cropIntended": "Maize",
    "terms": {},
    "documents": []
}
```

Request schema:

- `clusterId`: required string
- `budget`: required positive number
- `durationMonths`: required positive integer, max 600
- `startDate`: optional datetime string
- `cropIntended`: optional string, max 200 chars
- `terms`: required object record
- `documents`: optional string array

Response:

```json
{ "proposal": { "id": "uuid", "status": "SUBMITTED" } }
```

Side effects:

- Sets proposal status to `SUBMITTED`.
- Notifies the primary representative, or first representative if no primary exists.

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `400 INVALID_BODY`
- `404 CLUSTER_NOT_FOUND`

### 7.6 `GET /api/proposals`

Auth: required.

Purpose: list visible proposals for the current user.

Query parameters:

- `status?: DRAFT | SUBMITTED | UNDER_NEGOTIATION | ACCEPTED | REJECTED | WITHDRAWN`
- `clusterId?: string`

Visibility rules:

- `ADMIN`: all proposals.
- `REPRESENTATIVE`: proposals for clusters where the user is a representative.
- `INVESTOR` and other non-admin roles: proposals where `investorId` matches the current user.

Response:

```json
{
    "proposals": [
        {
            "id": "uuid",
            "status": "SUBMITTED",
            "budget": "50000.00",
            "durationMonths": 12,
            "startDate": null,
            "cropIntended": "Maize",
            "documents": [],
            "terms": {},
            "createdAt": "2026-05-01T00:00:00.000Z",
            "updatedAt": "2026-05-01T00:00:00.000Z",
            "cluster": { "id": "uuid", "name": "Cluster name" },
            "investor": { "id": "uuid", "name": "Investor Name" }
        }
    ]
}
```

Errors:

- `401 UNAUTHENTICATED`
- `400 INVALID_QUERY`

### 7.8 `POST /api/proposals/:id/submit`

Auth: required.

Role: `INVESTOR` only.

Purpose: promote a draft proposal to `SUBMITTED`.

Validation:

- Draft must exist.
- Caller must own the draft unless caller is `ADMIN`.
- Draft must currently be `DRAFT`.
- Draft must be complete enough to submit:
    - budget > 0
    - durationMonths > 0
    - terms object must not be empty

Response:

```json
{ "proposal": { "id": "uuid", "status": "SUBMITTED" } }
```

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `400 INCOMPLETE_DRAFT`

### 7.9 `POST /api/proposals/:id/decision`

Auth: required.

Role: `REPRESENTATIVE` only.

Purpose: accept or reject a proposal.

Body:

```json
{ "decision": "ACCEPT", "reason": "Optional rejection reason" }
```

Body schema:

- `decision`: `ACCEPT | REJECT`
- `reason`: optional string, max 500 chars

Response:

```json
{ "proposal": { "id": "uuid", "status": "ACCEPTED" } }
```

State behavior:

- `ACCEPT` sets status to `ACCEPTED` and triggers agreement generation.
- `REJECT` sets status to `REJECTED` and stores `rejectionReason`.

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `400 INVALID_BODY`
- `403 ONLY_REPRESENTATIVE`

### 7.10 `POST /api/proposals/:id/withdraw`

Auth: required.

Role: `INVESTOR` only.

Purpose: withdraw an open proposal.

Response:

```json
{ "proposal": { "id": "uuid", "status": "WITHDRAWN" } }
```

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `403 ONLY_INVESTOR`

### 7.11 `POST /api/proposals/:id/revisions`

Auth: required.

Purpose: submit revised terms while the proposal is still open.

Body:

```json
{
    "budget": 60000,
    "durationMonths": 14,
    "terms": {},
    "note": "Optional note"
}
```

Body schema:

- `budget`: positive number
- `durationMonths`: positive integer, max 600
- `terms`: record object
- `note`: optional string, max 2000 chars

Allowed actors:

- Investor or representative who can access the proposal

Response:

```json
{ "proposal": { "id": "uuid", "status": "UNDER_NEGOTIATION" } }
```

Side effects:

- Updates terms, budget, duration.
- Sets status to `UNDER_NEGOTIATION`.
- Emits realtime `proposal:revised`.

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `400 INVALID_BODY`

### 7.12 `GET /api/proposals/:id/messages`

Auth: required.

Purpose: list negotiation messages for a proposal.

Query parameters:

- `before?: string` - cursor message id
- `limit?: number` - default 50, min 1, max 100

Behavior:

- Messages are returned in ascending order.
- Cursoring is based on `createdAt` of the `before` message when it belongs to the same proposal.

Response:

```json
{
    "messages": [
        {
            "id": "uuid",
            "proposalId": "uuid",
            "senderId": "uuid",
            "message": "Hello",
            "attachments": [],
            "counterTerms": {},
            "isRead": false,
            "createdAt": "2026-05-01T00:00:00.000Z",
            "sender": { "id": "uuid", "name": "Sender Name" }
        }
    ],
    "hasMore": true
}
```

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `400 INVALID_QUERY`

### 7.13 `POST /api/proposals/:id/messages`

Auth: required.

Purpose: post a negotiation message.

Body:

```json
{
    "message": "Counter offer text",
    "counterTerms": {},
    "attachments": ["https://example.com/doc.pdf"]

```

Body schema:

- `message`: string, 1-4000 chars
- `counterTerms`: optional object record
- `attachments`: optional array of URL strings, max 10

Allowed actors:

- Investor or representative involved in the proposal

Response:

```json
{
    "message": {
        "id": "uuid",
        "proposalId": "uuid",
        "senderId": "uuid",
        "message": "Counter offer text",
        "attachments": [],

        "isRead": false,

    }
```

Side effects:

- Emits realtime `negotiation:message`.
- Sends a notification to the counterparty when applicable.

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `400 INVALID_BODY`

- Agreement status must be `DRAFT` or `PENDING_SIGNATURES`.
- Duplicate signature is rejected.

Response:

```json
{
    "signature": {
        "id": "uuid",
        "agreementId": "uuid",
        "signerId": "uuid",
        "role": "INVESTOR",
        "signedAt": "2026-05-01T00:00:00.000Z"
    },
    "agreement": {
        "id": "uuid",
        "status": "PENDING_SIGNATURES"
    }
}
```

State behavior:

- If both parties have signed, status becomes `PENDING_SIGNATURES`.
- Otherwise it remains `DRAFT`.

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `409 ALREADY_SIGNED`

### 8.5 `POST /api/agreements/:id/cancel`

Auth: required.

Body:

```json
{ "reason": "Optional cancellation reason" }
```

Validation:

- Actor must be investor, representative, or admin.
- Agreement cannot already be `COMPLETED` or `CANCELLED`.

Response:

```json
{ "agreement": { "id": "uuid", "status": "CANCELLED" } }
```

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `400 INVALID_BODY`

### 8.6 `PATCH /api/agreements/:id/terms`

Auth: required.

Purpose: edit agreement terms and/or clauses before activation.

Body:

```json
{
    "terms": {},
    "clauses": [{ "title": "1. Parties", "body": "Clause text" }],
    "note": "Optional note"
}
```

Body schema:

- `terms`: optional object record
- `clauses`: optional array of `{ title, body }`
- At least one of `terms` or `clauses` must be present
- `note`: optional string, max 500 chars

Behavior:

- Allowed only while the agreement is `DRAFT` or `PENDING_SIGNATURES`.
- All existing signatures are deleted.
- Agreement status resets to `DRAFT`.
- A notification is sent to the other party and all representatives.

Response:

```json
{ "agreement": { "id": "uuid", "status": "DRAFT" } }
```

Errors:

- `401 UNAUTHENTICATED`
- `403 FORBIDDEN`
- `404 NOT_FOUND`
- `409 INVALID_STATE`
- `400 INVALID_BODY`

## 12) Realtime contract references

Realtime events are not HTTP endpoints, but several API responses trigger them.

Observed server-to-client event names include:

- `notification:new`
- `proposal:revised`
- `negotiation:message`
- `negotiation:read`
- `proposal:typing`
- `agreement:created`
- `agreement:signed`
- `agreement:terms-edited`
- `agreement:activated`
- `agreement:cancelled`
- `receipt:uploaded`
- `receipt:rejected`

These events matter because they are part of the user-visible contract even though they are delivered via Socket.IO.

## 13) Data-shape mapping to frontend types

The frontend API type definitions in `apps/web/lib/api/types.ts` reflect the contract returned by these routes:

- `ClusterSummary`, `ClusterDetail`
- `ProposalSummary`, `ProposalDetail`, `ProposalViewer`, `NegotiationMessage`
- `AgreementSummary`, `AgreementDetail`, `AgreementSignature`, `AgreementTerms`
- `PaymentReceipt`

When changing the server payloads, update the corresponding frontend types in lockstep.

## 14) Known gaps and cautions

1. `api/uploads` is referenced by the server bootstrap and frontend client, but no route file is present yet.
2. The auth contract is externally owned by Better Auth, so the repository only guarantees the proxy mount, not each individual generated subroute.
3. Some error shapes are not fully normalized; callers should inspect `error`, and optionally `issues`, `status`, `required`, or `missing`.
